## 1. 产品定位

> 一款面向**软件/IT 外包公司**（中小企业 10–200 人）的中文、多租户、按席位订阅的**云 SaaS**——本质是 **PSA（专业服务自动化）+ 轻 ERP**，以"**项目全生命周期 + 工时 + 实时盈亏**"为核心闭环。

**关键定性（来自需求澄清）**

| 维度 | 结论 |
|---|---|
| 目标用户 | 软件/IT 外包公司，10–200 人（中小企业）|
| 商业模式 | 对外销售的 SaaS 产品，**按席位订阅** |
| 项目形态 | 混合：长周期大项目 + 短工单 + 持续运维续约 |
| 数据基础 | 全新绿地，无历史数据迁移 |
| 部署 | 云 SaaS 单一托管（首站**阿里云 ECS**）|
| 规模 | 中小够用 + 预留扩展（不过度设计）|
| 开发方式 | 主要由 AI 编写，用户偏产品/业务 |
| 节奏 | 不赶时间、求质量（全程 TDD）|
| 语言/视觉 | 中文单语言；界面**极简现代**（Linear/Notion 调性）|

## 4. 架构

### 4.1 方案选择

对比 **A. TypeScript 全栈单体** / B. Java+Spring / C. Odoo 二开，选定 **A**。

**理由**：主要由 AI 编写、用户偏产品，"单一语言 + 端到端类型安全 + 清晰模块边界"最大化产出正确性与可维护性；前后端共享类型，后端结构变更前端即时报错。B 双语言样板拖慢迭代；C"卖 Odoo 壳"掌控力与授权皆是坑，与"自有产品"冲突。

### 4.2 形态

**模块化单体（modular monolith）+ pnpm monorepo**。中小规模不上微服务；模块边界在代码层清晰隔离，未来易拆分。

### 4.3 技术栈

| 层 | 选型 | 理由 |
|---|---|---|
| 后端 | NestJS (TypeScript) | 模块化 + 依赖注入，强制清晰边界 |
| 数据库 | PostgreSQL | 事务强、JSON 灵活、行级安全(RLS) |
| ORM | Prisma | 类型安全、迁移友好；中间件注入 `tenant_id` |
| API | REST + OpenAPI | 自动生成 client/文档；利于门户/集成/私有部署 |
| 前端 | React + TS + Vite | 主流稳妥 |
| UI | Tailwind + shadcn/ui + TanStack Table/Query + 图表 | 极简现代且扛数据密集 |
| 认证 | 自建账号(argon2) + JWT(access/refresh) + 租户上下文，预留 SSO | — |
| 异步任务 | BullMQ + Redis | 催收、燃尽快照、通知、报表预计算 |
| 文件 | 存储抽象接口，默认本地磁盘，预留对象存储 | 平滑切 OSS |
| 部署 | Docker Compose（app + PG + Redis）| 云 SaaS 单托管，后续可 K8s |
| 测试 | Vitest + Supertest + Playwright，全程 TDD | 求质量 |

### 4.4 多租户隔离策略（云 SaaS 单库）

- 每张租户数据表带 `tenant_id`；
- 请求级"租户上下文" + Prisma 中间件**自动注入**租户过滤（应用层强隔离）；
- 关键敏感表加 **PostgreSQL 行级安全(RLS)** 兜底（第二道防线）；
- 抽象层就位，未来可演进到"每租户独立库/私有部署"。

### 4.5 模块地图（bounded contexts）

```
platform      租户上下文 / 审计日志 / 存储抽象 / 公共
iam           租户·用户·认证·RBAC·组织架构(部门树)
crm           客户·联系人·商机
projects      项目·Backlog·Sprint·任务·看板·燃尽·里程碑·验收·变更
resourcing    员工资源画像·技能·产能·排期·请假考勤
timesheets    工时(计划排期 + 实际自动汇总)
costing       成本归集(人力/分包/采购/报销)·预算·实时盈亏
billing       报价·合同·计费引擎·发票·应收催收
workflow      通用审批引擎(状态机)
documents     文件管理·知识库/Wiki
notifications 消息中心(站内/邮件/IM)
reporting     经营看板·自定义报表

```

每模块统一 **Controller(API) → Service(领域逻辑) → Repository(Prisma)** 三层；跨模块通过**服务接口**或**领域事件**通信，避免互相纠缠。

## 5. 核心数据模型

### 5.1 基类约定

所有业务表继承：`id` · `tenant_id`（隔离键）· `created_at/by` · `updated_at/by` · `deleted_at`（软删除）。金额一律 **Decimal** + `currency`（默认 CNY）。

### 5.2 价值主线核心聚合

```
Customer ──< Opportunity ──(赢单)──> Project
                                        │
        ┌───────────────────────────────┼───────────────────────────────┐
        │                               │                               │
     Milestone ──< Acceptance      Sprint ──< Task ──> Assignee(Employee)   ChangeRequest
     (里程碑/验收，可挂开票)          (Backlog/看板/燃尽)   │                    (变更→影响评估→重报价)
                                                        │
                                    ┌───────────────────┴───────────────┐
                              ResourceAllocation                    TimeEntry
                              (计划工时: 人×小时排期)              (实际工时: 任务自动汇总, 可校正)
                                                                        │
                                                                   ×成本单价
                                                                        ▼
Project ──< CostEntry(type: 人力/分包/采购/报销)
Project ──  Budget(项目/组织)
Project ──< Revenue(按计费模式确认)   →  【实时盈亏】= 收入 − 成本
```

### 5.3 逐模块关键实体

| 模块 | 关键实体 |
|---|---|
| iam | `Tenant` · `User` · `Membership`(用户↔租户,含席位/状态) · `Role` · `Permission` · `Department`(自关联树) |
| crm | `Customer` · `Contact` · `Opportunity`(阶段/赢单率) |
| projects | `Project`(状态机) · `Sprint` · `Task`(estimateHours/状态/看板列) · `Milestone` · `Acceptance` · `ChangeRequest` |
| resourcing | `Employee`(成本单价) · `Skill` · `EmployeeSkill` · `Capacity` · `ResourceAllocation`(计划) · `Leave` |
| timesheets | `TimeEntry`(实际,归属 Task+Employee,自动生成可校正) |
| costing | `CostEntry`(统一四类) · `Budget`(项目/组织) · `PnLSnapshot`(盈亏快照) |
| billing | `Quote` · `Contract` · `Invoice` · `Receivable` · `Subcontract` |
| workflow | `WorkflowDefinition` · `WorkflowInstance` · `ApprovalTask` |
| 横切 | `Attachment`(多态) · `AuditLog` · `Notification` · `KnowledgeArticle` |

### 5.4 三个关键设计决策

**① 计划工时 vs 实际工时**——两条线，省填报又能算成本
- 计划：`ResourceAllocation`（人×小时×时段，喂产能利用率与预算）。
- 实际：`TimeEntry` 归属 Task+人，**系统按任务状态/进度自动生成**，成员/PM **一键校正**。→ 不用天天填日报，又能喂真实成本。

**② 统一成本归集**——一张 `CostEntry` 容纳四类
人力由 `TimeEntry × 成本单价` 自动生成；分包/采购/报销各生成一条。全部 `type` 标记、挂 Project（可细到 Task/Milestone）→ 统一滚动汇总。

**③ 实时盈亏**——读时计算 + 快照加速
`盈亏 = 已确认收入(按计费模式) − Σ成本`。中小规模读时实时算；经营看板用 `PnLSnapshot`（后台定时刷新）。`Budget` 与实际并排 → 预警。

## 6. 关键业务流程

### 6.1 项目全生命周期（状态机）

`立项Draft → 进行中Active → 交付中Delivering → 结项Closed`（+ 挂起/取消）。三种立项来源（商机转化/手工/门户提需求审核）。每次跃迁有守卫（权限 + 前置条件，如"未验收不可结项"）。

### 6.2 工时→成本→盈亏（事件驱动）

```
① PM 排期            → ResourceAllocation(计划工时)
② 成员推进 Task      → 系统自动生成/更新 TimeEntry(实际) → 一键校正
③ 事件 TimeEntryChanged → costing 监听 → 按成本单价生成 CostEntry(labor)
④ 分包/采购/报销通过 → 各生成 CostEntry(其余三类)
⑤ 盈亏 = 已确认收入 − ΣCostEntry （读时实时 + PnLSnapshot 加速）
⑥ Budget vs 实际 → 超预算 → 通知预警
```
模块间**只靠领域事件通信**（工时变了，成本模块自行重算，两者不互相 import）。

### 6.3 计费引擎（策略模式，4 模式）

统一 `BillingStrategy` 接口，一模式一实现，产出 `Invoice → Receivable`：

| 模式 | 触发/算法 |
|---|---|
| 固定总价分期 | 里程碑验收通过 → 触发该期开票 |
| T&M 工时 | 周期结算：Σ(本期实际工时 × **计费费率**) |
| 周期订阅 | 定时任务按月自动生成账单 |
| 按交付验收 | Acceptance 通过 → 触发开票 |

应收账期到期 → BullMQ 定时任务 → 催收提醒。

> **⚠️ 计费费率 ≠ 成本单价**：计费费率是"对客卖价"，成本单价是"内部人力成本"。二者分开存，毛利 = 计费 − 成本 才准——PSA 核心。

### 6.4 通用工作流引擎（状态机，与业务解耦）

```
业务触发(工时/报销/变更/请假) → 按 WorkflowDefinition 创建 WorkflowInstance
   → 生成 ApprovalTask → 通知审批人 → 处理 → 流转/驳回
   → 完成 → 发结果事件 → 业务模块监听回调(如"报销通过"→生成 CostEntry)
```
引擎只管流转、不懂业务；业务只管发起 + 收结果。一套引擎服务所有审批。

### 6.5 权限与租户隔离（运行时四层防护）

```
请求 → JWT → RequestContext{userId, tenantId, roles, dataScope}
  ├─ 租户隔离：Prisma 中间件自动注入 tenant_id + 关键表 RLS 兜底
  ├─ RBAC：操作级 Guard 校验 Permission（能不能做）
  ├─ 数据级：dataScope(全部/本部门/本项目/本人) → 注入查询条件（能看哪些行）
  └─ 字段级：成本/利润敏感字段 → 序列化层按角色脱敏（能看哪些列）
```
层层收窄，**默认拒绝**；租户隔离是最外、最硬一道。

## 7. 横切关注点

### 7.1 错误处理
- 统一异常体系 `DomainError/ValidationError/AuthzError/NotFound` → 全局 ExceptionFilter → `{code, message, details, traceId}`。
- 边界校验：外部输入在 API 边界用 DTO + zod/class-validator 校验并清洗。
- 事务：跨表写用 Prisma 事务；**领域事件在事务提交后发布**（outbox 思路）。
- 幂等：开票、审批回调等关键写操作带幂等键。
- 前端：全局错误边界 + 请求拦截器统一处理 401/403/5xx。

### 7.2 安全与审计
- 认证：argon2 哈希 + JWT(access 短/refresh 长 + 轮换) + 登出失效。
- 传输/静态：全站 HTTPS；ECS 云盘静态加密 + 敏感字段应用层加密。
- 审计日志：`AuditLog` 记录谁/何时/改了哪个实体/前后值，随领域事件落库，只追加不可改。
- 通用防护：限流、CORS 白名单、安全响应头、Prisma 参数化防注入、前端转义防 XSS、默认拒绝。
- 数据归属：租户可一键导出全部数据。

### 7.3 测试策略（TDD 硬门禁）
- 单元(Vitest)：领域服务、计费策略、盈亏算法、权限判定——纯逻辑重点覆盖。
- API(Supertest)：每端点，**必含"越权/跨租户被拒"用例**。
- E2E(Playwright)：主线 `立项→排期→工时→实时盈亏`、`报价→合同→开票→应收`。
- 节奏：红→绿→重构；每测试用独立租户隔离。

### 7.4 阿里云 ECS 部署与运维

```
单台 ECS ── Docker Compose
  ├─ nginx      反代 + HTTPS(Let's Encrypt/阿里云SSL) + 托管前端静态资源
  ├─ app        NestJS(先 1 实例)
  ├─ postgres   数据卷持久化 + 云盘加密
  └─ redis      BullMQ 任务 + 缓存
```
- CI/CD：GitHub Actions 构建镜像 → ECS `docker compose pull && up -d`。
- 配置：`.env` 注入，**密钥不硬编码**。
- 可观测：健康检查端点 + 结构化 JSON 日志（后续接阿里云 SLS）+ 基础告警。
- 备份：`pg_dump` 定时 → 本地 + 传 OSS。
- **域名走大陆需 ICP 备案（早办）**。
- 规模化平滑路径：PG→RDS、Redis→云数据库、文件→OSS、邮件→DirectMail、app 多实例 + SLB。
- 环境：本地开发(compose dev) + ECS 生产（"试试"阶段单机既做体验也做试点）。

|---|---|---|
| 增值税票对接 | 工程量大、合规复杂 | 隔离到 P3，先做账单 PDF |
| 工作流引擎 | 通用引擎是重头 | 模型 P1 预留，实现 P2；先支撑固定几类审批 |
| 计费费率维护 | 费率随角色/客户/合同浮动 | 费率表支持多维度覆盖（默认→角色→客户→合同）|
| 实时盈亏性能 | 大项目成本行多 | PnLSnapshot 定时刷新 + 读时增量 |
| 本地磁盘存储 | 单机磁盘扩展/备份有限 | 存储抽象层，规模上来切 OSS |
| ICP 备案 | 大陆访问硬门槛 | 立项即启动办理 |
